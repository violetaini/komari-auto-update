Automatic updater for binary installations of [Komari Monitor](https://github.com/komari-monitor/komari). It follows the same release-detection idea used by the Komari web panel: compare the local `/api/version` result with official GitHub Releases, ignore drafts and prereleases, then update only when a newer stable release exists.

The project is intentionally narrow: it updates the Komari server binary managed by systemd. It does not install Komari, does not manage `komari-agent`, and does not support Docker or Compose deployments.

## Features

- One-command installer for systemd-based binary Komari deployments.
- Checks official Komari GitHub Releases and skips draft/prerelease builds.
- Compares semantic versions before downloading anything.
- Downloads the exact release asset, not a moving `latest` URL.
- Backs up the current binary and `data` directory before replacement.
- Keeps only the configured number of backup versions.
- Rolls back automatically if the new service fails to start or version verification fails.
- Uses `flock` to prevent concurrent update runs.
- Uses a systemd timer with configurable interval.
- Provides a generated uninstall command.
- README badges use JSON generated by this repository's GitHub Actions workflow from GitHub's own API data.

## Supported Deployment

Supported:

- Linux
- systemd
- Komari installed as a local binary, usually `/opt/komari/komari`
- a Komari systemd service, usually `komari.service`

Not supported:

- Docker
- Docker Compose
- source-tree development deployments
- agent-only updates
- installing Komari from scratch

The installer verifies that the configured systemd service runs the configured Komari binary. If that check fails, it stops instead of guessing.

## Quick Install

```bash
curl -fsSL -o install-komari-auto-update.sh https://raw.githubusercontent.com/violetaini/komari-auto-update/main/install.sh && chmod +x install-komari-auto-update.sh && sudo bash install-komari-auto-update.sh
```

Default behavior:

- check interval: `6h`
- backup retention: `2`
- Komari binary: `/opt/komari/komari`
- Komari service: `komari.service`
- local version API: `http://127.0.0.1:25774/api/version`

## Install With Custom Frequency

Use `--interval` to change how often the updater runs. The value is passed to systemd `OnUnitActiveSec`.

```bash
sudo bash install-komari-auto-update.sh --interval 12h
```

Common examples:

```bash
sudo bash install-komari-auto-update.sh --interval 6h
sudo bash install-komari-auto-update.sh --interval 12h
sudo bash install-komari-auto-update.sh --interval 1d
```

You can also use environment variables:

```bash
INTERVAL=12h KEEP_BACKUPS=3 sudo -E bash install-komari-auto-update.sh
```

## Install With Custom Backup Retention

Use `--keep-backups` to control how many binary backups and data backups are retained.

```bash
sudo bash install-komari-auto-update.sh --keep-backups 2
```

The cleanup logic keeps the newest `N` binary backups and newest `N` data archives under:

```text
/opt/komari/backups/auto-update
```

## Custom Paths

If your binary, service, or local API path differs:

```bash
sudo bash install-komari-auto-update.sh \
  --bin /opt/komari/komari \
  --service komari.service \
  --api http://127.0.0.1:25774/api/version \
  --interval 6h \
  --keep-backups 2
```

## Usage

Run an update check manually:

```bash
sudo /usr/local/sbin/komari-auto-update
```

Dry run:

```bash
sudo /usr/local/sbin/komari-auto-update --dry-run
```

View timer status:

```bash
systemctl status komari-auto-update.timer --no-pager -l
systemctl list-timers --all | grep komari-auto-update
```

View logs:

```bash
journalctl -u komari-auto-update.service -n 80 --no-pager
journalctl -u komari-auto-update.service -f
```

Uninstall:

```bash
sudo /usr/local/sbin/komari-auto-update-uninstall
```

or:

```bash
sudo bash uninstall.sh
```

## Configuration

The installer writes:

```text
/etc/komari-auto-update.conf
```

Example:

```bash
KOMARI_BIN="/opt/komari/komari"
KOMARI_SERVICE="komari.service"
KOMARI_API="http://127.0.0.1:25774/api/version"
KEEP_BACKUPS="2"
```

To change the run frequency after installation, edit:

```text
/etc/systemd/system/komari-auto-update.timer
```

Then reload and restart the timer:

```bash
sudo systemctl daemon-reload
sudo systemctl restart komari-auto-update.timer
```

For example, change:

```ini
OnUnitActiveSec=6h
```

to:

```ini
OnUnitActiveSec=12h
```

To change backup retention after installation, edit:

```text
/etc/komari-auto-update.conf
```

Then run:

```bash
sudo /usr/local/sbin/komari-auto-update --dry-run
```

## Safety Behavior

The updater is designed for unattended operation:

- exits when Komari is already on the newest stable version
- refuses to update when the Komari service is stopped
- refuses unsupported CPU architectures
- refuses releases without a matching Linux binary asset
- backs up the binary before replacement
- archives the `data` directory before replacement
- starts Komari after replacement and verifies the reported version
- rolls back binary and data backup if startup or verification fails
- removes old backups according to the configured retention

## Badge Data

The README badge JSON files under `.github/badges` are generated by `.github/workflows/update-badges.yml`. The workflow uses GitHub Actions and GitHub's official API data for this repository, then commits the generated JSON back to the repository.

The badges are rendered by `img.shields.io/endpoint`, but the statistics data is produced by this repository's GitHub Actions workflow rather than a third-party statistics service.
